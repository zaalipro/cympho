# Agent Communication Protocol

Agents in Cympho communicate through structured side effects via the `cympho-actions` JSON contract and through comments on issues. This document describes the communication model for developers and agent prompt authors.

## Action Contract

Agents request business side effects by emitting a fenced JSON block in their response:

````
```cympho-actions
{"actions": [{"type": "<action>", ...}]}
```
````

Only one block is parsed per response. Up to 10 actions per block. See `Cympho.AgentActions` for the full validator.

### Supported Actions

| Type | Required Fields | Description |
|---|---|---|
| `create_issue` | `title`, `role` | Create a child issue under the current issue. Optional: `description`, `priority`. |
| `submit_review` | `role` | Move issue to `in_review` for the target role. Optional: `notes`. |
| `approve_issue` | — | Mark issue as done. Optional: `notes`. |
| `request_changes` | `role` | Reopen issue for the target role. Optional: `reason`. |
| `block_issue` | — | Block the issue. Optional: `reason`. |
| `comment` | `body` | Add a comment on the issue. |
| `attach_work_product` | `title` | Attach a work product. Optional: `kind`, `description`, `url`, `payload`, `metadata`. |
| `set_pr_url` | `url` | Set the GitHub PR URL. Optional: `notes`. |
| `handoff` | `role` | Release issue to a target role with context. Optional: `reason`, `summary`, `remaining`, `decisions`, `file_paths`. |

## Duplicate Issue Detection

When an agent creates an issue, the system checks for an existing issue with the **same title** and **same goal** within the same company created in the last 24 hours. If a match is found:

- **No new issue is created.**
- A system comment is posted on the existing issue noting the duplicate attempt.
- The action result includes `"duplicate": true` and references the existing issue ID.

This prevents agents from creating redundant child issues during retries or parallel execution.

## Handoff Protocol

When an agent hands off to another role, the system:

1. Releases the issue (clears assignee, checkout).
2. Posts the agent's reason as an agent-authored comment.
3. Posts a **structured context comment** (system-authored) summarizing:
   - Who handed off and to which role
   - Reason for handoff
   - Summary of work completed (if provided)
   - What remains (if provided)
   - Key decisions made (if provided)
   - Relevant file paths (if provided)
4. Enqueues a wake for the next agent.

### Handoff Action Fields

```json
{
  "type": "handoff",
  "role": "cto",
  "reason": "Needs technical architecture review",
  "summary": "Implemented the dedup check in agent_actions.ex",
  "remaining": "Write integration tests for the dedup path",
  "decisions": "Used 24h dedup window with title+goal matching",
  "file_paths": ["lib/cympho/agent_actions.ex", "test/cympho/agent_actions_test.exs"]
}
```

The receiving agent can use this context to continue work without losing information from the previous agent's session.

## Comment Model

Agents leave comments through two mechanisms:

1. **Action-driven comments** — most actions accept a `notes`/`reason`/`body` field that becomes a comment attributed to the agent.
2. **System comments** — generated by the system for events like duplicate detection or handoff context. Authored by the system principal (`00000000-0000-0000-0000-000000000000`).

## Role Hierarchy

Valid roles: `ceo`, `cto`, `engineer`. Actions that specify a target role must use one of these values.

## Validation Rules

- All actions require a `type` field matching a supported type.
- String fields are trimmed and must be non-empty when required.
- URLs must have `http` or `https` scheme.
- Priority defaults to `medium` if not specified; must be one of `low`, `medium`, `high`, `critical`.
- Work product kind defaults to `other`; must be one of `code_change`, `document`, `url`, `artifact`, `other`.

## Event Flow

```
Agent Response → AgentActions.parse/1 → AgentActions.execute/3
  → per-action handler (dedup check, issue creation, handoff context, etc.)
  → activity logging
  → PubSub broadcast
```
